Problem Definition

            Markov models (extensions of Markov chains) are useful tools that mimic logic-level state machines. These models consist of a number of states where probabilities govern transitions between them. In this project, we are performing language processing, having the computer recognize an input sentence and determine if it is reasonable English. In our case, we are considering a Hidden Markov Model (HMM). An HMM has numerous states, transition probabilities, and output/observation probabilities. The states, in this case, are unobserved or hidden and only information about state transition and output observation is available. We are tasked with building an HMM using the forward/backward procedures, the Viterbi algorithm, and the Baum-Welch algorithm for optimization, to recognize a given sentence from a small 8 word dictionary and determine its validity.

Method and Implementation

            We are provided with a file .hmm which describes the structure of the model we will implement. From the file we have extracted the structure, pictured below.

Given the values for N, M, T, English syntactic structures, the possible output words, a matrix (state transitions) b matrix (output probabilities), and pi vector, we can go about deciphering input sentences. The observations, or outputs, are written to a file .obs which holds the number of observations followed by the actual output. The program generates the probability that these results are acceptable and reports it to the console.

 

            Our code structure consists of the model constructed by the .hmm file, the input file handler that is the second argument, and the functions that analyze the input file and perform probability calculations. First, we parse the .hmm file such that the input equations from the Rabiner paper can be converted into code (the model is lambda.) This is the first argument of running the program. The second argument is the observation file, which holds the sentences that the model analyzes. When the program runs with the Viterbi algorithm, it outputs probabilities according to how valid the sentence is. This recognizes if it is an acceptable sentence based on the sentence structure.

(QUESTION 2 ANSWER)

            In our model, we can extract the optimal path using statepath.java. This generates an output like the 4 below.

kids play food                    0.036 SUBJECT PREDICATE OBJECT

 

eat kids food robots       4.0000000000000013E-4 PREDICATE OBJECT OBJECT OBJECT

 

can kids eat chess                0.004725 AUXILIARY SUBJECT PREDICATE OBJECT

 

kids eat food                     0.036 SUBJECT PREDICATE OBJECT

This describes the output of the most probable path taken. Here we get the most probable final state, then find the most probable state to have preceded it. This happens recursively. If there is more than one way to construct a sentence, it gives you the most likely way, such that it gives you the best output possible. The model, in our case, can distinguish between questions and statements by checking if AUXILLIARY is used to start a sentence. This is because the state transition sequence directly corresponds to the grammar composing the sentences.

 

            Optimizing the HMM utilizes the Baum-Welch algorithm, which calculates the probabilities both before and after it determines an optimal state sequence. BW algorithm makes use of the forward and backward procedures to update the initial state probability vector, the state transition matrix, and the resulting observed matrix and finally calculate an optimal path. The final, updated model, is exported to sentence-opti.hmm

 

(QUESTION 3 ANSWER)

            Optimizing an HMM with observation probability == 0 is erroneous because we cannot divide by 0. The BW algorithm includes calculations that would require division by 0, making this completely impossible by nature.

 

Conclusions

Our implementation also performs each of the algorithms fine, demonstrating the effectiveness of using such systems for language recognition and processing.

Exploring Neural Net Implementation Using the Back-Propagation Algorithm

CS 440 P 2
Peter Galvin and Max Porter

Problem Definition

               Training a Neural Net is one of the most fundamental aspects of implementing an artificial intelligence system. Like a weighted graph in mathematics, a NN has nodes and weighted lines connecting those nodes. Each set of nodes is a column vector known as a layer, and in our case, we have an input layer, multiple hidden layers and an output layer. The propagation of data through the net generates an output that dictates how we will change the weights of each connection. After the weights are changed, we repeat this process of input, weight change, and output to minimize the error between the calculated value and expected value. This is an iterative algorithm known as back-propagation, and is used to train neural nets for numerous applications and data sets.

 

Method and Implementation

               We are given skeleton code that constructs the neural net as a set of vectors corresponding to the different layers, where each node (element) in these column vectors is connected via a weighted line. This idea is implemented in the classes Node.java and Connection.java. These are then combined into a neural net via the NeuralNet.java object. Using this construct, we built a train function in NeuralNetLearner.java which reads a set of data, particularly the credit or lens data, and adjusts weights based on beta values. This affects the error after each iteration, which is hopefully minimized after enough rounds. We had to normalize the input data by creating a normalize function in the data processor class. There were unique normalization functions for both sets of data.

               A walkthrough of the training algorithm behaves the following way. To start, using a given vector of inputs, we calculate an output by initially passing through the net using the randomly generated weights. The calculated output is found by use of the NeuralNet function. The function calcOutput then takes the output of each layer, and feeds them to the next layer as a set of new inputs. This is how the propagation occurs. Computation of Beta then takes place at the end of the first pass through the neural net at the output layer. Beta is then calculated for the previous layer, and the layer before that. This is the essence of back propagation. Once Beta is found for each node including threshold nodes, their respective delta ws are computed. This process is repeated for each input vector; the last input vectors delta w is finally calculated and all are aggregated. This aggregated delta w is then implemented to change the weights.

To do this, we split our experimentation into separate trials. We wanted to see the effect of different values of r and the number of iterations/rounds had on the error and the error rate for the training data and test data. Discussion

               The Neural Network we put together performed faster with fewer nodes, but generated a lower error and error rate the more nodes we added to the hidden layer. This is expected, as more nodes makes the loss function more complicated, and therefore fits the data better. We used the structure with 30 nodes shown above to generate the results for the different r values and number of rounds. It must be noted that we used the same z normalization process for both the Credit data and lens data. As shown in the Credit Data training results, having a small learning rate paired with many rounds diminishes the error. The test data, on the other hand, shows that any learning rate below 1, regardless of the number of iterations, remains stagnant at about the same values. Because the test data is significantly smaller, its more difficult to approximate a trend.

               The lens test data provided erroneous results. We expected to receive very small errors and error rates of 0, but instead received a consistent value of 0.5. The training data, however, gave us the expected results. This inconsistency could be with how we preprocess the data, but this was a recent development and given the time constraint, was not addressed.

 

Conclusions

Given a large enough training data set, a neural net can adjust itself to provide an output of minimal error. The limitation of the net, however, is that the data must be interpreted correctly in order for the net to respond. This limits the use of neural nets and makes them unique to the type of data it is analyzing.

Introduction

Computer vision systems are used for actions such as playing games and recognizing some environment. Products like the XBOX Kinect use image analysis to enable the user to perform gestures in space, taking their actions as input and allowing them to control games that appear on a screen. We are tasked with a similar idea: design computer vision code that is able to recognize certain actions within a captured video frame.

For P1, we were asked to design a vision system that recognizes gestures performed in front of a web cam. The gestures Max and I worked to implement included drawing small dots with a fist, drawing bigger dots with one's palm, erasing by showing the edge of the hand, and erasing everything by waving a hand.

Available Gestures

Our system recognizes these gestures by grabbing the camera feed, converting to an HSV image (hue, saturation, value) and filtering skin by using a set threshold. Once the binary image is made and skin is filtered, we can find the centroid of this object in the frame by using the built in CV functions. This gives us attributes like area, position in (x,y) of the hand, and distance of the centroid between frames. Using these properties, we create a Mat type painting (from the image class of P0) where we transfer the coordinates of the centroid to from the filtered image.

In order to differentiate between when the user draws a small dot, bigger dot, or wishes to erase, we compare the circularity of the blob to a known template of a circle. Depending on the roundness of the user's gesture, we assign a value that corresponds to the likelihood that the user is making a fist, or has their palm exposed. For detecting the edge of the hand, a similar method is used, but this time we look for a more oblong or ovular shape, and compare that.

Our system is slightly more robust than those which use the RGB colorspace. We convert to HSV to account for more lighting situations. Regardless of the colorspace, however, keeping a set filter consistent between environments is ill advised. Because people will use this perhaps in the lab, as a demonstration in a classroom, or in their room, the filter that was set for some specific environment wont work in another. Thus, for the program to work reliably, we assume the user is in an area where the background is far from skin tone, well lit, and somewhat static (not much moving.)

Assumptions and Issues

Given the limitations outlined above, the system is obviously not very robust, and there is room for improvement. The ability to calculate the centroid and localize it is hard coded, and we do not account for multiple objects; thus the average of all pixels within a range gives us an erroneous x,y position. Using a ball remedies this somewhat, as it is a more easily filtered object, making a large, contiguous blob. The program accurately draws small and large dots and also erases locally and the whole image quite reliably.

Discussion/Future Work

Our vision system works quite well given the size of our team and the inconsistency of background. We can reliably obtain the position of a yellow ball and translate that to paint displayed on a new image frame.

Right now our system just takes the centroid calculated by blob filtering and converts that x,y position to a pixel location on a new frame. This isn't really painting, and is more like the spray paint can in MSPaint. One way we can go about drawing lines is using Bresenham's march algorithm, or better known as Bresenham's line algorithm, to connect each point/dot drawn for each centroid calculation. This creates the ability for the user to draw consistent lines, making the idea of painting that much more effective!
